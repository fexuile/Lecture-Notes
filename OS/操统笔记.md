## Introduction
**什么是操作系统？** 
> 一些特殊的提供到硬件资源的软件层。  
> 1. 作为“魔术师”，提供简单、干净的物理资源的抽象。
> 2. 作为“裁判”，保证公平和共享之类的技术。
> 3. 作为“胶水”，实现一些简单的服务。


**操统做什么？** 
> 1. 对应用程序提供抽象（比如说文件系统之类的）
> 2. 管理系统资源
> 3. 实现一些特殊的算法和技术

实际上操作系统就是实现一些high-level的硬件实现，处理进程公平和资源共享等问题，合并了计算机科学的诸多领域。

## Scheduling
调度策略的目的是为了：
1. 最小化完成时间
2. 最大化吞吐量
3. 保证公平

CPU burst：CPU 运行程序指令的时间

### Policy
- FCFS：First Come，First Server。先进先出（FIFO）策略。
- Round Robin（RR）：将时间划分成若干个小slice，对于花销小的任务比较友好，但是时间开销大的任务需要花费更多的时间在上下文切换上。核心点就是**抢占**
- SPS：Strict Priority Scheduling 维护若干个优先级队列，然后从最大的开始选取。可能造成饥饿的问题，以及优先级上导致的死锁。
- SJF（STCF）： Shortest Job First 
- SRTF（SRTCF）： Shortest Remaining Time First 非常的优秀，但是很难预测未来情况，以及不公平。由于未来非常难以预测，所以我们可以采用一个近似的方法来预测当前任务的长度（$\tau_n = F(t_{n-1},t_{n-2}, \dots) = \alpha t_{n-1} + (1-\alpha)\tau_{n-1}$ 
- Lottery Scheduling： 为了近似SRTF，我们给short-time的分配多一些彩票，long-time的少一些，然后在CPU里面抽签选择。
- Multi-Level Feedback Scheduling（MLFS）： 一般来说我们在比较高priority的队列里用RR（可以用多个队列表示不同的RR粒度），在低priority里的用FCFS。然后如果timeout了就降级，周期性的将所有的任务放置在最高级的队列中。
- EDF： Earliest Deadline First 选取ddl时间最早的任务完成。
- LCFS： Last-Come，First-Served

Priority Scheduling可能会导致死锁，所以我们需要一个priority donation。详细的就像lab里面写的一样做就好了。

Nice Scheduling：用一个expired和一个active队列来存储任务，然后动态地调整他们的priority。nice值越高，优先级越低，反之。

CFS： Completely Fair Scheduler 选择当前调用cpu时间最少的进程（类似于堆的调度），保证几乎完全的公平。在multi-core的情形下，不同处理器的$Q_i$可以通过不同的$weight$来划分。

### Deadlock
死锁出现的条件：资源独占、等待、没有抢占、循环等待。
资源图之中存在一个环并不代表存在死锁，必须得不断地循环来探测。
**Deadlock Detection Algorithm:** 我们定义三种数组，
- $FreeResources$:表示每种资源还剩下多少未分配的
- $Request_x$：表示Thread x对于每种资源还剩下多少的request
- $Alloc_x$：当前Thread x每种资源占用了多少
然后我们对于每一个Unfinished的thread考虑能否进行分配所有资源(request <= Avail)，如果可以的话就把他Finish，然后将他的$Alloc_x$放置到$FreeResources$里，不断重复该过程直到无法新Finish。

我们解决死锁有如下方式：
- 阻止死锁：让你的code不会触发死锁，让你这个Thread需要的锁在一开始就都申请，按照一定顺序去申请锁（或者给申请锁的操作加一个锁操作）。
- 死锁恢复：采用类似回滚的做法（参考数据库中的transaction，但是如果restart的顺序有问题还是会导致死锁）。
- 避免死锁的发生： 动态地延迟资源请求，我们可以添加一个unsafe state。
- Deadlock denial：忽略死锁的可能性。

然后这个unsafe的状态侦测我们只需要和资源图的死锁侦查的判定条件改成Max-Alloc就可以了。
即：Request <= Avail -> Max - Alloc <= Avail。
其中的Max表示该线程需要某种资源的最大数量的数组。

## Memory
比较基础的是直接映射，然后可以采用base&bound限制。base&bound可以通过offset+base的方法来进行最简单的地址翻译。但是这个会导致碎片化的问题，以及share会非常的困难
然后比较高级一点的是采用segment的方法进行地址翻译，类似于\[EX:...\]的样子（体系讲过），但是这个方法的上下文切换开销太大了（更换进程的时候需要换出它的segments），也会存在很多的外部/内部碎片。
更加nb的是采用页表的形式，用一个page table来进行翻译到物理页号，offset直接继承。
优点是便于共享，上下文只需要切换一个指针，问题在于页表需要的存储空间比较大，开销比较多，所以我们引入了多级页表。

然后多级页表我们可以采用多个PTE，也可以第一个采用seg的形式，第二个采用Page Table的形式。x86-64用的则是4级页表（4\*9+12）
多级页表的优点是便于分配，便于共享；但是缺点就是查询消耗了大量的时间，以及页表需要连续。

最后一种则是采用hash表的办法，将每一个虚拟页号映射到一个物理页号。优点是Page Table的大小和实际的物理页数目基本一致。

为了加速地址翻译，我们添加一个TLB（采用全相联，因为conflict的代价非常大），前面的VPN就作为标志，后面的offset分开作为组号和偏移量。分为：全相联（s=1），直接映射和组相连三种。全相联的冲突miss是最少的。

直写和写回：写回的优势在于时间开销小，缺点在于复杂、read miss会导致修改数据的写回。

在TLB之中我们实际上还会加入一些Access rights的位置，来进行protection等任务。

如果我们发现当前某个我们需要的物理页不在，我们就需要触发一个Page Fault，然后进行物理页面的替换。实际上这个过程和PTE的过程非常的相似，只不过是发生在DRAM上。实际上Demanding Page就是等同于做了一个cache，对应的cache空间叫做内存。

如同cache有计算access time, Demanding Page也有：“Effective Access Time” = Hit_rate \* Hit_time + Miss_rate \* Miss_time。

页替换策略也有很多种：
- FIFO
- RANDOM
- MIN： 替换将来不会被用到的时间最长的页，是optimal的，但是这个要求知道未来的状况。
- LRU：替换距离上一次使用时间最远的页，一般来说应用中采用approximate LRU.
如果在ABCDABCD...(一共3个Page)的情况下，LRU和FIFO是等价的，不如MIN的表现。

我们考虑将需求页面抽象成若干个working set, 以miss的数量作为纵轴，page数量作为横轴画图：随着page的增加，LRU和MIN的miss会减少，但是FIFO不一定（例子是ABCDABEABCDE，一共5个page）.

我们考虑页面替换的近似LRU算法：
- Clock Algorithm：用一个指针在Memory Set之中寻找，如果一个use=1，就设置为0，否则就决定evict这个页面。他的进阶是Nth-Chance...，如果一个use=0了，就把他的count递增，第一个count=N的就是被替换的页面。注意Dirty页面应该额外给一次机会，因为他有额外的替换负载。
	- 我们不需要额外的硬件支持实现use和modify bit，这都是可以通过OS来实现的。
- Second Chance List Algorithm：把内存分成两个list：active list和SC List，其中Active采用FIFO（被逐出的页面进入到SC List里，SC List采用LRU算法）。如果产生的Page Fault页面在SC里面，就把他从SC到active，如果都不在那么就直接放到Active里面。比FIFO的优点是更少的disk access，但是更大的trapping overhead。
对于Clock Algorithm有一个额外的优化，新建一个free list，然后和Active Set之间的关系就像Second Chance List一样。脏页面的换出在CPU空闲的时候会进行，增加效率。注意，这个新算法（free list）不改变Active Set的状态，是只读的。

发生页面evict之后要更新指向这个页面的pte

我们对于frame的分配，可能有Fixed分配和Priority的分配，低Priority的帧可能被高的征用。Fixed的话有两种，一个是都固定相同的size，另一个则是根据process的大小来分配size。

我们可以动态的调整一个process得到的page的size，设定一个lower rate一个upper rate，如果miss rate比lower rate低就增加size，比upper rate高就增加size。

Thrashing是指CPU的大部分开销都在swapping page之中，导致利用率急剧下滑。这个时候我们定义page reference序列上的一段为working set，然后其中的不同的page数目为他的大小，$D = \sum_{i=1}^kWS_i$,如果$D$超过一个数量就会导致Thrash。

有了working set的概念之后，我们可以对冷不命中做一些操作,比如说Track Working Set，或者利用locality来提前将附近的page加入到memory之中。

### 一些论文
RDMA: Remote Direct Memory Access. 现在做大模型就需要去购买NVIDIA的全家桶

- FaRM: 新建一个Share Address space，然后我们可以将一系列的数据放在同一个主机上减小数据传输的时间开销，然后在这台计算机上面进行计算。最后还有一个对事务（Transaction）的优化，首先是Execution，然后lock，再验证，最后update和unlock。**关键点就是提供了共享内存的概念。**
- InfiniSwap: 如果一个机器没有足够的空间去分配，我们将其他机器空闲的内存放置在一个Memory Disaggregation Layer之中，然后remote调用这个内存。实际上就是通过网卡进行swap，也应用了RDMA的技术。Infiniswap Block Device的映射如果粒度很细就会导致Table很大，所以我们将页大小变大一点，粒度变粗，使得查找的代价变小。里面有Power of Two，就是随机取两个然后选择其中剩余多的进行映射。
- AIFM：重新实现了若干个数据结构（如数组，队列等）的API，有4个核心的科技：
	- 只访问特定数据而不是访问整个页，以解决语义的gap
	- 用userspace runtime直接在用户态之间切换，避免kernel的overhead。
	- Pauseless evaluator：将object从local移动到remote的时候不需要阻塞，而是可以同时进行。
	- Remote Agent：一些简单轻量的操作直接传输到remote的machine去处理。
上述三者，FaRM对Application进行了较大的更改，得到了更好的performance；Infiniswap不需要更改App的实现，但是性能不及FaRM；AIFM则是在二者之间达到了一个平衡。
- PipeSwitch：提升模型训练时将不同的模型切换到GPU Memory之中的性能。在数据传输和数据计算中做一个pipeline（为了解决同步等问题采用分组的办法），在内存分配上采用一个统一管理的daemon，我感觉类似Central Controller。任务初始化的时候我们将Initialize分成两部分，一部分是加载，另一部分是内存分配空间；然后先把第一部分都做了，在switch的时候一边clean之前的一边alloc新任务。
- TGS：**Key ideas: leverage CUDA unified memory to transparently unify GPU memory and host memory**，主要的就是一个类似于lazy-load的模式以及当GPU Memory oversubscribed的时候，采用evict的方法给更高优先级的任务分配资源。
PipeSwitch对于应用而言不是transparent的，而TGS是的。
## I/O
对于IO设备我们可以采用端口号映射也可以采用直接映射到物理内存地址（Memory Mapping的优点在于可以protect）。将数据传输到Controller的办法有两种，一个是Programmed I/O，优点是简单，缺点是时间开销大；另一个则是DMA（Direct Memory Access）的办法去访问内存bus。
I/O devices的分类有：
- Block Device：比如说磁盘、DVD等。
- Character Device：比如说keyboard。
- Network Device：Bluetooth等。
这其中有blocking、unblocking和异步三种等待方式。

对于磁盘，重点介绍一些Track Skewing，就是说每一个Track的开头并不是在同一个位置，这是为了让访问更加的efficient。
对于SSD，读就是若干个颗粒块，读非常的快，但是写和擦除很慢，都以1个数量级的方式递增。
所以可以采用Copy-on-write的技术以及Layer of Indirection（也就是说如果我们一个很小的改动我们就直接擦除修改而是直接找一个新的地方map）。
关于SSD和磁盘的访问时间：
- 磁盘（HDD）：Queue + Controller + Seek + Rotational + Transfer
- SSD： Queue + Controller + Xfer
关于piazza上面的一个提问，Disk Request的(a,b)分别表示cylinder和sector，cylinder是track相同的一个柱面，所以等于是定了track和sector。Seek就是针头去放置在不同的track上。

衡量设备性能的两个标准：Latency和Throughput，延迟和吞吐量。
我们建立一个数学模型来表示things的处理：
Arrival -> Queue -> Serve -> Departure.
每隔$T_A$时间段就会来一个任务，需要排队$T_Q$的时间，用$T_S$的时间Serve。
那么利用率就是$U = \frac{T_S}{T_A}$，同时我们定义$\mu$和$\lambda$为Service rate和Arrival Rate.
新定义一个变量C表示方差平方系数，为$\frac{\sigma^2}{E^2}$。E是期望，$\sigma^2$是方差。
如果$C=0$则是常数,$C=1$则是无记忆分布。Disk的respone time一般为C=1.5.

然后由于指数分布的特点，我们可以知道Service Time和Arrival Time的期望就是$\mu$和$\lambda$。
我们希望得到的参数分别是Time of Queue, Length of queue. 由一个叫做**A Little Queue Theory的理论**可以得到，$T_q = T_S \times (C+1)\frac{u}{2(1-u)}, L_Q = T_Q * \lambda$。

Disk Scheduling：因为一次只能处理一个Disk Request，所以他们之间的处理顺序非常的重要：
- FIFO：公平，但是很长的seek time。
- SSTF： Shortest Seek Time First， 可能会导致饥饿（注意这里要计算rotation time，因为他们差不多需要消耗同样的时间）。
- SCAN：用一个类似电梯的方法来扫描整个序列，就不会出现饥饿的情况，但是还是不够公平。
- C-SCAN：电梯可以从1楼直接到最高层，保证了公平性。

## Filesys
Logical Block Addressing (LBA)：给每一个sector一个编号，就可以从地址翻译到Disk对应的位置。
我们采用File name -> File number的一个map来进行Filename的translation，然后实际上整个的结构就是：
File name ->(directory structure)  File number ->(index structure) storage block.
inode实际上就是一个file对应的metadata（我们也可以叫做file header）。
### 经典的file system
#### FAT
**FAT：FILE ALLOCATION TABLE**，一个存储Disk Allocation情况的Link，你只要给定File number就能够顺序的遍历这个file对应的blocks。注意，FAT之中还存储了Access Right等数据，FAT整个存储在Disk上。
**root directory一般来说会被存储在Block2，因为没有Block0和Block1**

#### UNIX FILE SYSTEM
从inumber->inode，inode之中存储了metadata。同时，inode之间构成了一颗树型结构。
前面的若干个是direct block，每个block的大小是1KB，后面的几个分别是indirect，double和triple，通过ptr数量和block大小可以计算不同level的block的存储信息大小。
注意在BSD4.1之中block size是1KB，但是到了之后的就是4KB了。

#### BSD4.2 Fast File System
FFS利用了locality，引入了一个Block Groups的概念，也就是说把inode和整个目录下的file存储在同一个cylinder group之中，减少seek的时间。
注意到这有一个问题，就是我们在读取之后做process的时候rotate还在进行，可能会错过一些信息，可以通过pre-read的办法，或者我们本身在磁盘中存储就隔着一定的间隔。
优点：
- 对于小文件来说，没有seek的时间。
- 可靠性更强。
- 没有碎片重整。
缺点：
- 对于小文件而言利用率低，要用一个group来存储inode和data。
- 当文件在磁盘上大部分是连续的时，编码效率低下（不是很懂）
- 要在Block Group之中预留10%的空间避免碎片化
Link是构建出File Tree的一个关键，有Hard Link和Soft Link(Symbolic Links)
两种，Unix通过symlink来实现soft link，通过构建文件的时候就新增link关系，link和unlink两种指令来新增或者remove link关系。

对于一个比较大的目录，如果我们还是顺序遍历的话会非常的慢，所以我们可以采用B树的方法（Hash）。
#### WINDOWS NTFS
NTFS：New Technology File System
使用了一个Master File Table，然后文件的大小不是Fixed的，而是由很多个extent块组成。
对于比较小的文件就直接存储在MFT之中，也可以存储A list of extents，甚至可以是指向另一个MFT Record的指针，其中有更多的extents list。MFT的Record之中存储了file name，file number等。

Mmap就是把一个File的映射到user address之中，这样子可以实现share的功能。

Kernel之中用了一个Buffer cache来存储kernel resources，有disk blocks和name translation，那些没有被置换的脏块也会存储在这里。这个功能完全是由操作系统实现的， 

一些思路：
- Filesystem的cache替换策略可以是LRU，也可以是只使用一次（使用完就丢弃）。cache的size最好是可以动态调整，因为如果很大的话app就没有足够的空间了，如果很小的话他们就会运行得很慢。
- Filesys可以预取，因为一般来说process会顺序的读取一系列的文件。但是预取多少是一个问题，预取太多了就会导致其他app request的delay过高，太少了就会导致seek时间过长。
- buffer cache采用延迟写回的思路，只有在buffer filled的时候，或者buffer被刷新的时候才会写回。还会定期地进行写回，为了保证不会出现crash错误。

文件系统最为重要的三个方面：能力（指对请求的接受和操作能力）、耐用性（指可以recover）、可靠性。

而RAID则是提升了文件系统的耐用性，每个disk都有一个copy备份，写就写两次，读就从任意一个disk读。如果出现了failure就从备份中恢复。
RAID-5指的是对于每5个做一个效验码，然后把他们放在不同的Disk上，保证如果存在一个disk fail了还能够恢复。
**Reed-Solomon codes**则是更为常用的，$n$个数据生成$m$个效验码，可以容忍m个错误。（也就是说$n+m$个中随便选$n$个都能够计算出剩下的$m$个）

关于可靠性的提升，有两种办法：
1. Careful Ordering and Recovery：就是谨慎地将操作序列排序，保证能够从每一步recovery。如果有data和指向data的指针，我们应该先修改data再修改指针。FAT,FFS,许多应用(emacs, Word)都采用这么办法。
2. Versioning and Copy-on-Write：WAFL、ZFS/OpenZFS都是采用的这个办法。实际上当你修改inode的一个结点的时候，只需要像主席树一样操作就好了。
3. 使用事务(transaction)这个新的abstract：将一些对于文件的操作atomicly，具体的做法为：开始（begin），做操作（如果中间产生错误或者冲突就回滚roll-back），最后提交（commit）。采用日志的方法来提升可靠性。
	- **Log Structured：数据被存放在log之中。**
	- **Journaling：log是用来恢复的。**

### 分布式系统
从构想之中来说，分布式系统可以满足更大的availability，更加的durability，以及more security.但是实际上来说则是恰恰相反。
Two-Phase Commit Algorithm：有一个协调员来询问所有的worker是否可以工作。如果都Vote Commit那么就会Commit，只要有一个Abort就会Global Abort。
具体的步骤就是协调员询问worker，发出VOTE—REQ，工人返回VOTE-COMMIT/VOTE-Abort，然后再由协调员发出统一指令Global-COMMIT/Global-Abort.
每个节点都使用稳定的存储（如SSD/NVRAM）来记录当前状态。

### 一些论文
- Dedup：重复数据消除。核心技术在于：
	- Summary vector：测试某个数据是否存在（如果存在过一定为1，为1不一定存在）
	- Stream informed segment layout：把同一个stream中的segments放在一个container之中，可以类比Block Group的实现。
	- Locality preserved caching（LPC）：把metadata用一个cache来存储。
- IOFlow：需要系统能够实现end-to-end的SLAs。定义了一个Control plane，在data plane之中实现了controllable queue来实现SLAs。同时，对于每一个要求，我们用一个流来抽象（VM，File op， Destination）->SLA. 用这些把IO分到不同的queue之中，每一个queue有不同的优先级以及做法。
	- 在每一层都翻译成这一层理解的格式，这样子就可以传输SLA。
	- token_rate：不能直接把token和IO简单绑定，而是要根据真实的存储开销。建立请求cost的模型（将大的请求切分成若干个小请求，然后把queue和对应的cost模型绑定即可）
	- 在分布式的情况下，动态调整，采用Max-min fair sharing。
- GFS：Google File system，有一个GFS master，是中心化的single master，Client把读取发个master，然后master把需要读取的数据发给存储数据的distributed chunk servers，最后由servers发给Client。
- EC-Cache：核心在于Reed-Solomon code，然后将他们放在不同的machine上面。可以实现负载平衡以及降低读入的延迟。
- Chord：p2p相关，我们将服务器ip进行hash然后摆放在一个环上面，对于需要访问的数据也做一个hash，将数据存储在最靠近它的下一个服务器节点上，最后用一个类似倍增的做法寻找数据；关于插入一个新服务器只需要修改后继节点的数据信息即可。
